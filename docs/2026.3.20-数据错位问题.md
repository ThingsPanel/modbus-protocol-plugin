# 数据错位问题分析与解决方案

## 一、问题现象

同一网关下多个寄存器（如 `switch` 和 `temp`）的采集数据发生错位：
- 期望：读开关返回 `{"switch":1}`，读温度返回 `{"temp":25.6}`
- 实际：某些请求超时后，下一个请求的响应被错误匹配到之前的请求上

## 二、根本原因分析

### 当前超时配置

- 写超时：1 秒
- 读超时：1 秒

### 为什么现有串行锁不能解决问题

代码中 `DeviceRWLock[regPkg]` 互斥锁**已经是按设备串行的**，锁的范围覆盖"发送请求 + 读取响应"整个过程。同一个设备内，请求是串行的。

但问题的根源不在于"并发发请求"，而在于**迟到响应不受锁保护**：

```
T0: goroutine-A 发送读开关请求，设置读超时1秒
T1: goroutine-A 等待响应，超时（T1 - T0 > 1秒）
T2: 开关的响应实际到达（迟到），躺在连接的内核缓冲区里
T3: goroutine-A 释放锁，返回超时错误
T4: goroutine-B（下一个采集周期）拿到锁，发送读温度请求
T5: goroutine-B 读取响应 → 读到的是 buffer 里 goroutine-A 的迟到响应（功能码0x01而非0x03）
结果：温度寄存器收到了开关的值 → 数据错位
```

**内核缓冲区中的数据不受用户态互斥锁控制**，这是核心问题所在。

### 为什么"等一下再发"不够

"错误后等一段时间再发下一个请求"的思路是让设备把积压的旧响应发完。但设备有自己的发送节奏，网关无法控制设备何时停止发送。等多久都不够——设备可能更慢。

## 三、最佳方案：超时后断开连接

### 原理

断开 TCP 连接时，操作系统的 TCP 栈会向设备发送 FIN/RST 包，**强制设备侧清空 TCP 发送缓冲区**。设备重连后，所有状态干净，不会有任何历史残留数据。

```
问题场景（不重连）：
设备发送旧响应 → buffer 里永远有脏数据 → 后续请求可能被污染

方案（断开重连）：
网关关闭连接 → 设备 TCP 栈收到 FIN/RST → 设备清空 buffer → 设备重连 → 干净状态
```

### 实施步骤

#### 步骤 1：`handleRTUCommand` 和 `handleTCPCommand` 中，超时不重试

**改动点**：`services/handle_conn.go` 中的 `handleRTUCommand` 和 `handleTCPCommand` 函数。

当检测到超时错误时，不进行重试，也不继续下一个采集周期，而是**立即关闭连接，让设备重连**。goroutine 退出后，`verifyConnection` 会处理设备的重连流程，采集自动恢复。

#### 步骤 2：`verifyConnection` 中，设备连接后先清空残留

**改动点**：`services/services.go` 中的 `verifyConnection` 函数。

设备连接后（包括重连），先清空 TCP 连接缓冲区中的残留数据，再处理注册包，防止上次断开时的残留数据干扰。

## 四、与其他思路的对比

| 方案 | 效果 | 副作用 |
|------|------|--------|
| 延长超时时间 | 无效，每次采集都变慢 | 所有采集延迟增加 |
| 错误静默期（等一下再发） | 有限，设备节奏不受网关控制 | 治标不治本 |
| 超时断开重连 | **最彻底，强制清空设备 TCP buffer** | 该网关下所有子设备短暂离线（毫秒级） |

## 五、副作用及应对

断开重连会影响该网关下所有子设备的采集，造成短暂离线。但：
- 离线时间通常只有几百毫秒，设备重连后采集自动恢复
- 相比数据错位导致的永久性数据污染，短暂离线是可接受的代价
- 现有的 `flushTimeoutResponse` 排空机制作为最后一道安全网保留
